üöÄ 50% Application Performance Overhaul

Achieved a 50% Boost

Application Performance Improvement for a Public Sector System.

SQL Server Database Migration System Integration

The Problem

  • Critical bottlenecks causing 30-second query waits during peak reporting times.
  • Legacy indexing strategy resulting in unsustainable high I/O and CPU utilization.
  • Risk of data integrity failures due to poor system load handling.

The Solution

A multi-phase database and application re-architecture focusing on deep-level performance tuning, index optimization, and **data flow modeling** to ensure robust, scalable access for over 1,000 public users.

The Process: Deep Dive into Re-Architecture

We used **SQL Server Profiler** and **Query Store** to identify the top 10 most resource-intensive stored procedures. The analysis showed that $85\%$ of the load was caused by two critical reporting joins with high I/O waits.

Code Proof: Example of Costly Join
-- Original, non-indexed query taking >15 seconds
SELECT * FROM Users u JOIN Orders o ON u.UserID = o.UserID; 
-- Execution cost: 85% Table Scan on Orders table

We implemented a revised indexing strategy, prioritizing **Clustered Columnstore Indexes** on large, fact-based tables to improve I/O efficiency. We also refactored the two identified bottleneck queries, replacing subqueries with highly-optimized Common Table Expressions (CTEs).

Visual Proof: Before vs. After Execution Plan

The solution was deployed in a staging environment for User Acceptance Testing (UAT) with 50 concurrent users. We also audited the new database connections to ensure compliance with HIPAA security standards, specifically focusing on **encryption-in-transit** protocols.

Quantifiable Results

+50%

Faster Application Performance

-20%

Reduction in Server CPU Utilization

0

High-Priority Data Integrity Alerts Post-Deployment

Key Takeaway

The biggest lesson was the value of focusing on **clustered index architecture** before looking at costly hardware upgrades. Smart re-architecture negated the need for a $\mathbf{\$10,000}$ server purchase, proving that database design is the first line of defense against latency.

Reclaimed Team Hours

Developed automated scripts to eliminate manual software updates and plug a critical security gateway.

Automation (AutoIT) Process Improvement Security

The Problem

  • Manual updates (Adobe, Java) consumed **10+ hours per week** of high-level IT staff time.
  • Update delays left endpoints vulnerable to common exploits.
  • Inconsistent patching led to compliance and audit risks.

The Solution

Developed an **AutoIT scripting program** paired with a central control file to automate the download, validation, and installation of critical third-party software across all systems, ensuring rapid and consistent security coverage.

The Process: Scripting and Implementation

The core program used AutoIT to interact with Windows processes and GUIs. It read a network-shared **INI file** to determine which software needed updating and the required version, providing a centralized control point for the deployment team.

Proof: Example INI Configuration

[Adobe_Reader]
Version=2024.002.20698
DownloadURL=//server/patches/Adobe.exe
Checksum=A2C3F4D5E6

The script included logic to verify the installer **checksum** against the INI file before execution, ensuring the integrity of the patch. The process logged success or failure to a central database, providing real-time auditability for all patched endpoints.

This strategy secured the deployment pipeline and eliminated the human error inherent in manual patching.

Quantifiable Results

~40 Hrs/Mo

IT Staff Time Reclaimed from Manual Patching

100%

Endpoint Compliance with Latest Patches

AutoIT

Primary Technology Utilized

Key Takeaway

This project reinforced the principle that **automation is the single most effective security measure** for routine maintenance. By investing development time upfront, we not only plugged a critical exploit gateway but also gained an entire person-week of productivity every month.

üó∫Ô∏è GIS Data Model Transformation

First-Time Infrastructure Mapping

Engineered a Python script to translate legacy tax data into a clean, spatial model for city infrastructure management.

Python Data Modeling POSTGIS/GIS

The Problem

  • Public Works infrastructure assets (pipes, conduits) were **unmapped** and lacked spatial data.
  • Core reporting database had a complex **one-to-many relationship** for tax lots and addresses, unsuitable for GIS joining.
  • Legacy data architecture prevented efficient spatial querying and reporting.

The Solution

Developed a custom, external **Python script** to consume the multi-relation data set, restructure it into a simple **one-to-one logical model**, and integrate the cleaned data directly into the ArcGIS environment for mapping and asset management.

The Process: Data Restructuring and Integration

The central task was translating relational data into a flat, spatial-friendly format. The **Python script** utilized database connectors (like `psycopg2` or `pyodbc`) to extract and merge related address records into a single row per tax lot, resolving the one-to-many conflict before ingestion.

Code Concept: Core Data Merging Loop

# Conceptual loop to merge multi-row addresses into one record
for lot_id in unique_lots:
    address_list = get_addresses_by_lot(lot_id)
    merged_address = " / ".join(address_list)
    update_gis_table(lot_id, merged_address)

The cleaned data was imported into ArcGIS, forming the foundation for the Public Works layer. This enabled the development of several **custom GIS tools** (written in Python/ArcPy) to automate labeling, feature selection, and reporting based on the newly available spatial data.

This project successfully paved the way for the City of Newport to include all subsequent infrastructure management within their GIS framework.

Key Takeaway

This project proved that effective **data modeling and transformation** is the critical first step in leveraging powerful applications like GIS. Without the custom script to flatten the data, the entire infrastructure mapping initiative would have stalled at the integration phase.
